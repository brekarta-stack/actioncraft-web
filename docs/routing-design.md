# 모델 로스터 + 라우팅 규칙 설계서 (LiteLLM 기준)

> 원칙: 워크플로는 **역할 별칭(alias)** 만 호출한다. 실제 모델·폴백·예산은 라우터가 결정한다.
> 버전 표기는 대표값이며, 실제 설정 시점에 각 제공사 최신 버전을 확인해 넣을 것.

---

## 1. 등록할 모델 로스터 (후보 확정)

### 클라우드 (API 키 필요, 종량제 — Claude Code 구독과 무관)

| 역할 | 모델 | 담당 | 대략 단가 감각 |
|---|---|---|---|
| 프리미엄 A | **Claude (Sonnet 주력 / Opus 최난도)** | 대외 발행물 최종 퇴고, 매매 분석, 견적·계약 문안, 최난도 코딩 | 고 |
| 프리미엄 B | **OpenAI GPT (상위 티어)** | 프리미엄 A의 대체·교차검증(second opinion) | 고 |
| 중간 저가 | **Kimi K2.x (Moonshot 직접)** | 대량 배치 중 품질이 필요한 것, 프리미엄 폴백 1순위 | 저 ($0.6/$3 per M 급) |
| 초저가 빠름 | **GPT-mini 또는 Gemini Flash** | 값싼 대량 처리, 간단 분류의 클라우드 폴백 | 매우 저 |

### 로컬 (키 불필요, 전기값만 — 무제한·대기 없음)

| 역할 | 모델 | 머신 | 상시/온디맨드 |
|---|---|---|---|
| 코딩 주력 | **Qwen3-Coder-30B-A3B** (Q4, ~31GB, ~84 tok/s) | Studio 96GB | 상시 로드 |
| 코딩 고성능 | **Qwen3-Coder-Next 80B-A3B** (Q4, ~61GB, ~37 tok/s) | Studio 96GB | 필요 시 JIT 로드 |
| 한국어 산출 | **Qwen3.6-35B** 또는 **A.X-4.0-Light**(둘 다 Apache 2.0) | Studio 96GB | 필요 시 |
| 상시 소형 | **Qwen3.5-9B** (Q4, ~7GB) | Mini 24GB | 24/7 상시 |
| 임베딩 | **BGE-M3** | Mini 24GB | 24/7 상시 |

> 한국어 개인 학습 전용에 한해 EXAONE-32B 사용 가능(비상업 라이선스라 사업 산출물엔 금지).

## 2. 라우팅 별칭 표 (워크플로가 부르는 이름)

각 별칭 = "1순위 → 폴백 → 폴백" 체인. 앞이 죽거나 예산 초과면 자동으로 다음으로 내려감.

| 별칭 | 1순위 | 폴백 1 | 폴백 2 | 월 예산캡(예시) | 용도 |
|---|---|---|---|---|---|
| `classify-fast` | 로컬 Qwen 9B (mini) | Kimi | GPT-mini | — (거의 공짜) | 태깅·분류·리드 1차 스코어링 |
| `summarize` | 로컬 Qwen 9B/35B | Kimi | — | 소액 | 아카이브 요약, 짧은 요약 |
| `code-fast` | 로컬 Qwen-Coder-30B | Kimi | GPT-mini | 소액 | n8n 노드·스크립트·표준 리팩터링 |
| `code-heavy` | 로컬 Qwen-Coder-Next | Claude Sonnet | GPT | 중 | 다중파일·복잡 디버깅(내부용) |
| `code-max` | Claude (Opus/Sonnet) | GPT | — | 중 | 매매 로직 등 최난도·고신뢰 코딩 |
| `write-ko-draft` | 로컬 Qwen3.6/A.X | Kimi | — | 소액 | 블로그·SNS **초안** (한국어) |
| `write-ko-final` | **Claude** | GPT | — | 중 | **대외 발행물 최종 퇴고** |
| `bulk-quality` | Kimi | GPT-mini | 로컬 35B | 중 | 대량인데 품질도 필요한 배치 |
| `analyst-trading` | **Claude** | GPT | — | 중 | 주식 분석 리포트(읽기 전용) |
| `quote-legal` | **Claude** | GPT | — | 소액 | 견적서·계약 문안 |
| `embed` | 로컬 BGE-M3 (mini) | — | — | 0원 | RAG 임베딩 |

**규칙 요약**: 파란 글씨(대외·돈·신뢰)는 무조건 프리미엄 1순위. 나머지 대부분은 로컬 1순위. 예산 소진 시 자동 강등.

## 3. 정교화 규칙 (단순 배정을 넘어서는 4가지 장치)

### (1) 자동 승급 (self-escalation) — 로컬이 실패하면 한 단계 위로
로컬 모델의 코드 산출물을 **검증**(문법 체크, 테스트 실행, 스키마 검증)한 뒤, 실패하면 자동으로 한 티어 위 별칭으로 재시도한다.
- 예: `code-fast`(로컬) 결과가 테스트 실패 → 자동으로 `code-heavy` → 그래도 실패 → `code-max`.
- 무한 재시도 금지: **최대 2회 승급 후 사람(#approvals) 호출.**

### (2) 예산 게이트 — 80% 경고 / 100% 강등
별칭별 월 예산캡을 건다. 사용액 80% 도달 시 #ops에 경고, 100% 도달 시 그 별칭의 1순위(프리미엄)를 **자동으로 폴백 모델로 강등**해 월말까지 무중단 유지. → "토큰 충전 대기" 문제를 구조적으로 제거.

### (3) 프라이버시·민감도 게이트 — 민감 데이터는 로컬 우선
계좌 정보·개인 식별정보·미공개 사업 데이터가 포함된 요청은 **대외 노출이 아니어도 로컬 모델 우선**. 클라우드로 나가야만 하면 민감 필드를 마스킹 후 전송.

### (4) 시간대·자원 게이트 — 장중엔 mini 보호
장 시간(KRX 09:00–15:30, 미국 22:30–05:00 KST)에는 mini의 대량 임베딩/요약 배치를 미뤄 매매 엔진 지연을 보호한다. Studio는 `code-heavy`/`write` 호출이 오면 깨우고, 유휴 시 모델 자동 언로드(콜드스타트 대비 타임아웃은 넉넉히).

---

## 4. 판단은 최소로 — "규칙 90% / 에이전트 10%"
별칭 배정의 대부분은 "작업 종류만 보면 정해지는" 고정 규칙이다(위 2번 표). 매번 LLM에게 "어디로 보낼까?"를 묻지 않는다. 정말 애매한 소수(예: 이 리팩터링이 로컬로 될지)만 에이전트 판단에 맡긴다 — 판단 자체의 비용·지연을 아끼기 위함.

---

## 5. 셋업 순서 (실제 적용 시)
1. API 키 발급: **Claude, OpenAI, Moonshot(Kimi)** 3종을 LiteLLM에 등록.
2. 로컬 백엔드 등록: Studio(LM Studio 헤드리스 또는 Ollama)와 mini(Ollama)를 OpenAI 호환 엔드포인트로 LiteLLM에 연결.
3. 위 2번 별칭들을 `model_list`로 정의하고, 각 별칭에 `fallbacks`와 `max_budget`을 건다.
4. n8n·매매엔진·학습 파이프라인은 이제 실제 모델명 대신 **별칭만 호출**하도록 작성.
5. LiteLLM 관리 UI에서 별칭별 지출을 관제(§관제 시스템과 연결).

> 다음 단계로, 이 표를 그대로 구현한 **LiteLLM `config.yaml` 초안**(model_list + fallbacks + budget)을 생성할 수 있음.
