# 종합 운영 에이전트 설계도 (3-Tier Architecture)

> 목적: 15인 / 3개 사업 운영을, "소수 인원 + 에이전트"가 분담하는 체제로 전환.
> 핵심 원칙: **두뇌는 클라우드(Claude), 손발·상시성·대량처리·보안은 로컬 맥**으로 분담.
> "쉽고 양 많은 일은 아래에서 0원으로, 어렵고 가치 큰 일만 위(Claude)로 에스컬레이션."

최종 업데이트: 2026-05-29

## 2. TIER 0 — 두뇌 (Claude · 클라우드)

| 항목 | 내용 |
|---|---|
| 역할 | 오케스트레이터. 복잡한 판단, 멀티스텝 자율 실행, 최종 검수, 어려운 코딩 |
| 모델 | Claude Opus / Sonnet |
| 접속 | Claude Code (Max 구독) 또는 API |
| 특징 | **컴퓨팅 파워와 무관** — 추론은 전부 클라우드. 어떤 기기에서도 동일 성능 |

원칙: 전체 작업의 5~15%(어렵고 가치 큰 일)만 여기로 올린다. 나머지는 아래에서 처리해 토큰 비용·레이트리밋을 절약.

## 3. TIER 1 — 상시 호스트 (Mac Mini M4 Pro 24GB)

24시간 켜져 있는 "신경계". LLM 추론보다 **상시성·연결·라우팅**이 핵심 역할이라 모델 성능과 무관하게 제값을 한다.

- **자동화 런타임**: n8n / 크론 / 커스텀 오케스트레이터 상시 구동
- **MCP 도구 호스팅**: 6개 커넥터를 상주시켜 항상 호출 가능 상태 유지
- **웹훅 수신기**: 견적폼·Gmail push·Slack 이벤트를 새벽/주말에도 즉시 수신
- **작고 빠른 모델 (8~14B, MLX, Q5)**: 1차 분류·라우팅·트리아지·즉답 (14B Q4 ≈ 30+ tok/s)
- **로컬 벡터DB / 작업 큐 / 상태 저장**: RAG 인덱스와 워크플로 상태 보관

> 메모리 제약: 24GB 중 OS 제외 ~16~18GB. 8~14B가 적정, 32B는 빠듯, 70B는 무리. → "두뇌"가 아니라 "교통정리"용.

---

## 4. TIER 2 — 로컬 처리 레이어 (Mac Studio M3 Ultra 96GB) ★성능 극대화

대량·민감·무거운 작업을 토큰 비용 0으로 처리. 96GB 통합메모리 + ~800GB/s 대역폭을 최대한 활용하는 방법:

### 4.1 핵심 전략: MoE를 써라 (가장 큰 레버)

- MoE(Mixture-of-Experts)는 **전체 파라미터를 메모리에 다 올리되, 토큰당 일부 전문가만 활성화**한다.
- 결과: **메모리는 많이 쓰지만(96GB 통합메모리가 강점) 연산은 가벼워 속도가 빠르다.** = "큰 모델의 품질 + 작은 모델의 속도".
- Apple Silicon의 통합메모리 구조와 궁합이 가장 좋은 모델 형태다.
- dense 70B(Q5, ~48GB)는 12~18 tok/s에 그치지만, 활성 파라미터가 적은 MoE는 **같은/더 나은 품질에 30~50+ tok/s**가 가능.

### 4.2 모델 후보 (96GB 기준, 우선순위 순)

| 모델 유형 | 전체/활성 파라미터 | 대략 용량 | 특징 |
|---|---|---|---|
| gpt-oss-120b급 MoE | ~117B / ~5B active | MXFP4 ~63GB | 활성 5B로 빠름 + 컨텍스트 여유. **균형 1순위** |
| Mixtral 8x22B | 141B / ~39B active | Q4 ~80GB | 품질 좋음, 96GB에 빠듯하게 적재 |
| Qwen3-30B-A3B급 MoE | 30B / ~3B active | Q5 ~22GB | 매우 빠름·가벼움, 긴 컨텍스트. **속도 1순위** |
| dense 70B (Hermes 등) | 70B / 70B | Q5~Q6 ~48~56GB | 안정적이나 느림. MoE 대비 비효율 |

> 권장: **속도가 필요한 상시 작업은 소형 MoE, 품질이 필요한 배치 작업은 대형 MoE**로 2종 운영.

### 4.3 추론 엔진 (하드웨어 효율 극대화)

- **MLX / MLX-LM (Apple 네이티브) — 1순위.** 통합메모리에 최적화돼 Mac에서 GGUF보다 빠르고 메모리 효율적. `mlx_lm.server`로 OpenAI 호환 API 제공.
- **llama.cpp (Metal 백엔드)** — 성숙·광범위한 모델/양자화 지원(IQ quant, KV cache 양자화). MLX 미지원 모델 대비용.
- **LM Studio / Ollama** — 위 엔진들을 감싼 운영 편의 레이어(GUI/CLI, 서버 모드).
- vLLM/TGI는 CUDA 중심이라 Mac엔 부적합 → 사용하지 않음.

### 4.4 추가 가속·효율 기법

- **Speculative decoding**: 소형 드래프트 모델 + 대형 타깃 모델 조합 → 1.5~3x 가속 (llama.cpp/MLX 지원).
- **KV-cache 양자화 (q8/q4)**: 긴 컨텍스트에서 메모리 폭증 억제 → 더 큰 모델/더 긴 문맥 가능.
- **IQ 양자화 (importance-matrix)**: 같은 용량에서 더 높은 품질 유지.
- **Prompt/KV 캐시 재사용**: 반복되는 시스템 프롬프트 재처리 생략 → 에이전트 루프 가속.
- **Continuous batching (서버 모드)**: 에이전트의 병렬 호출 throughput 향상.
- **메모리 헤드룸 20~30% 확보**: KV 캐시 + OS 여유. 무리하게 큰 모델을 꽉 채우지 말 것.

### 4.5 TIER 2 담당 작업

사내 문서 임베딩/RAG 인덱싱(대량·반복), 대량 분류·요약·태깅·1차 초안, 회의 녹취 전사·OCR·배치 처리, **외부 반출 금지 민감 데이터** 처리.

## 7. AI 결과물을 슬라이드/워드로 받기

현재 연결된 커넥터 기준, **바이너리 파일(.pptx/.docx)을 Slack 창에 직접 업로드하는 도구는 없음.** 대신 다음 경로가 가능:

| 형태 | 경로 | 어디서 받나 | 비고 |
|---|---|---|---|
| 리치 문서(보고서/노트) | **Slack Canvas** (`slack_create_canvas`) | Slack 창에서 바로 (모든 기기) | 표·체크리스트·헤더 지원. 가장 즉시성 높음 |
| 슬라이드 | **Google Slides** (Drive `create_file`) | Drive 링크를 Slack에 공유 | 어디서든 열람·편집·발표 |
| 워드 문서 | **Google Docs** 또는 실제 `.docx` 업로드(Drive, base64) | Drive 링크를 Slack에 공유 | `.docx`/`.pptx` 원본도 Drive 업로드 가능 |
| 실제 .pptx/.docx 파일 | TIER 1/2에서 `python-pptx`/`python-docx`로 생성 → Drive 업로드 → 링크 | Drive 링크 | 사내 양식·완성도 높은 산출물용 |

- **Claude에서 직접**: 코드 실행 환경에서 실제 `.pptx`/`.docx` 파일 생성 가능(아티팩트/다운로드).
- **Slack에서 바로**: Canvas는 네이티브로 즉시. 실제 Office 파일은 "Drive 생성 → Slack에 링크" 흐름이 현재 구성상 가장 매끄럽다.

### 💰 경제
| 항목 | Claude 단독 | 이 구조 |
|---|---|---|
| 토큰 소비 | 단순 작업까지 전부 유료 | 단순·대량은 로컬 0원, Claude는 5~15%만 |

| 하드웨어 | 불필요 | 이미 산 맥 = 매몰비용 회수 |

### 🔒 그 외
- 프라이버시: 민감 데이터는 Studio에서만 처리(외부 미전송)
- 레이트리밋 독립: 대량 작업이 Claude 한도에 안 걸림
- 오프라인 폴백: 인터넷/API 장애 시에도 Mini 기본 기능 생존
- 데이터 주권: RAG 인덱스·업무 상태가 내 하드웨어에 잔존

### 단점(정직하게)
- 복잡도 ↑: 장애 지점 1개 → 3개. 셋업·유지보수 시간 필요
- 초기 구축 노력: 자동화 런타임·MCP 상주화·모델 서빙 세팅
- 라우팅 기준을 잘못 잡으면 로컬이 어려운 일을 떠맡아 품질 저하

## 10. 효율 최적화 — Hermes 하이브리드 확정 구성

헤르메스로 TIER 1·2가 이미 구현돼 있으므로, 남은 핵심은 **"어디서 무엇을 돌릴지" 분리**다.

1. **두 맥의 역할 분리 (가장 큰 효율 레버)**
   - **Mac Mini = 헤르메스 데몬(상시 호스트)**: 24시간 켜둬도 전력·발열이 작은 Mini가 오케스트레이션 + 라우팅 + Slack 수신 + 스킬 저장 담당. 무거운 모델은 올리지 않음.
   - **Mac Studio = 로컬 모델 서버**: MLX로 MoE 모델을 OpenAI 호환 엔드포인트로 서빙. Mini의 헤르메스가 LAN으로 호출. 필요할 때만 풀가동 → 평소 전력 절약.
   - 효과: 상시성(Mini)과 무거운 추론(Studio) 분리 → 응답성↑ + 전력↓.
2. **명시적 라우팅 정책 (비용 핵심)**: 기본값 = 로컬 모델. ①복잡한 다단계 판단 ②로컬 신뢰도 낮음 ③대외 최종 산출물일 때만 Claude로 에스컬레이션. → 토큰 비용 대부분 0원화, Claude는 5~15%.
3. **스킬 파일 = 복리 자산**: 헤르메스 자기개선 스킬을 git으로 버전 관리, 사업별 네임스페이스 분리. 반복 업무를 의도적으로 태워 스킬화.
4. **사업 3개 분리**: Slack 채널/컨텍스트를 사업별로 분리(`#biz-a-ops` 등)해 맥락·스킬 혼선 방지.
5. **산출물(슬라이드/워드)**: "pptx/docx 생성 → Drive 업로드 → Slack 링크/Canvas" 흐름을 헤르메스 스킬 1개로 박제.
6. **보안 (사업 운영 필수)**: 도구 권한 최소화 + 시크릿 분리 + 외부 스킬 검증 후 도입. 민감 데이터는 Studio 로컬 모델로만.

> 한 줄: **Mini(상시 라우터·헤르메스) → Studio(MoE 로컬 추론) → 어려운 것만 Claude.**

## 13. 안정화 우선 재설계 (2026-07-20) ★현행 우선 지침

> 실운영 결과: 현 구성은 **Slack – Hermes – Mac Mini – NAS**. 관리 요소 과다 + 간헐적 오작동 + Mini 관제 부담.
> 이 섹션은 위 §10~12의 "최적화 우선" 지침보다 **우선한다**. 최적화는 안정화 다음이다.

### 13.1 시스템의 실체

| 부품 | 실체 |
|---|---|
| Slack | 입출력 창구 |
| Hermes | 수신→판단→실행 루프를 도는 런타임 |
| Mac Mini | 24시간 호스트 |
| NAS | 데이터·백업 저장소 |

에이전트 = "메시지 수신 → 맥락 조립 → 모델 호출 → 도구 실행 → 결과 관찰 → 반복 → 응답"의 **단일 루프 프로그램**.

### 13.2 왜 잘 안 되는가

루프의 각 단계(수신/맥락/모델/도구/응답)가 전부 독립 고장 지점. 각 단계 95% 신뢰 × 5단계 = **전체 ~77%**. 부품이 아니라 **연결 고리 수와 불가시성**이 문제. 대표 고장: Slack 토큰 만료, 데몬 미기동(재부팅 후), 자동 생성 스킬 오염, 로컬 모델 서버 다운, NAS 마운트 해제.

### 13.3 설계 원칙

> **"운영자 1명이 관리할 수 없는 시스템은 없는 시스템이다."** 설계 = 기능을 관리 가능한 수준까지 줄이는 것.

```
[인터페이스]     Slack 채널 2개만: #agent(명령) / #agent-log(관제)
[오케스트레이션]  Hermes @ Mini — 라우팅·루프만. 채널/기능/스킬 최소화
[지능]           모델 1개로 시작(Claude) → 안정화 후 로컬 추가
[저장]           NAS(데이터) + git 레포(설정·스킬 = 단일 진실원)
```

- **정정**: 불안정 단계에서는 "기본 로컬, 어려운 것만 Claude"(§10.2)를 적용하지 않는다. **모델을 Claude 하나로 줄여 변수 제거** → 루프 안정화 후 로컬을 비용 절감 레이어로 복귀.
- **관제 역전 (pull→push)**: 들여다보러 가지 않는다. 모든 실패·생존신호가 #agent-log로 온다. 대시보드 폐기.

### 13.4 실행 순서 (Minimal Viable Loop)

1. 전부 끄기 — 남기는 것: Slack 수신 → Claude 호출 → Slack 응답 한 줄기
2. 생존 보장 — launchd KeepAlive 자동 재시작 + 하루 1회 heartbeat → #agent-log
3. 실패 즉시 알림 — 모든 에러를 #agent-log로 push
4. **2주 무사고** 후 기능 1개씩 복원. 복원 기준: "고장을 5분 내 알 수 있나?" / "따로 끌 수 있나?"
5. 이후 로컬 모델 → NAS 심화 → 사업별 채널 확장

### 13.5 관리 항목 — 이 5개 외에는 관리하지 않는다

| # | 항목 | 방법 | 주기 |
|---|---|---|---|
| 1 | 생존 | heartbeat 확인 | 매일 10초 |
| 2 | 설정 단일화 | config·스킬 git 커밋 | 변경 시 |
| 3 | 스킬 위생 | 자동 생성 스킬 심사·삭제 | 주 15분 |
| 4 | 로그 | #agent-log 실패 알림 확인 | 주 15분 |
| 5 | 복구 가능성 | git clone + NAS로 재구축 테스트 | 월 1회 |
6. **(F) 나머지 2개 사업 온보딩** — 데이터·흐름 연결
