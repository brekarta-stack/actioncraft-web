# 개인 에이전트 오케스트레이션 시스템 — 재설계안 (v2)

**작성일**: 2026-07-22 · **투입 기간**: 2주 · **하드웨어**: Mac Studio M3 Ultra 96GB / Mac mini M4 24GB / UGREEN DXP4800 Plus NAS

---

## 0. 한 줄 결론

> **"슬랙 채널별 에이전트가 일을 하는 구조"를 폐기하고, "NAS 위의 워크플로 엔진(n8n)이 일을 하고, 슬랙은 보고·승인만 받는 구조"로 전환한다.** 하드웨어 3대는 전부 살리되 역할을 재배치하고, 관제 대시보드는 직접 만들지 않고 기성 도구 3개를 묶는다. 클로드코드는 **개발 도구로 강등**시키고, 운영 런타임에서 완전히 제거한다.

기존 구조 유지 여부 판정: **부분 폐기 + 재편** (하드웨어 ✅ 유지 / 슬랙 ✅ 역할 축소 유지 / 채널별 에이전트 ❌ 폐기 / 자체 관제 대시보드 ❌ 폐기 / 클로드코드 세션 의존 ❌ 폐기)

## 1. 기존 구조에 대한 적대적 검토

요청하신 대로 현재 구조("각 하드웨어의 AI 모델 + 미니 관제 대시보드 + 슬랙 채널별 에이전트 + 클로드코드 세션 프로젝트")를 방어 없이 비판적으로 검토한 결과입니다.

### 1-1. 슬랙 채널별 에이전트 = 실행 주체 구조의 근본 결함

2025~2026년 커뮤니티(OpenClaw/Clawdbot 계열 사용자들 포함)의 공통된 결론은 명확합니다: **채널별 챗 에이전트는 인터페이스로는 훌륭하지만 실행 백본으로는 부적합**합니다.

- **비결정성**: 챗 에이전트는 같은 요청에 다른 행동을 합니다. "매일 오전 9시에 리드 수집 → 블로그 초안 → 발행"같은 반복 업무는 결정적(deterministic) 워크플로여야 하며, LLM이 매번 계획을 다시 세우면 실패율과 비용이 모두 올라갑니다.
- **토큰 낭비**: 상시 대기하는 채널 에이전트는 컨텍스트 유지·대화 관리에만 상당한 토큰을 소모합니다. "토큰 최적화"라는 당신의 목표와 정면으로 충돌하는 구조입니다.
- **관측 불가능**: 채널 10개에 흩어진 에이전트는 "지금 뭐가 돌고 있고 뭐가 실패했는지"를 한눈에 볼 수 없습니다. 그래서 관제 대시보드를 따로 만들게 됐던 것인데, 이는 구조가 만든 문제를 도구로 덮은 것입니다.
- **보안 표면**: 항상 켜져 있는 채널마다 프롬프트 인젝션 표면이 열립니다. 특히 주식 매매 권한이 이 구조에 물려 있다면 심각한 리스크입니다.

### 1-2. 자체 제작 관제 대시보드(미니)의 문제

- 관제 대시보드는 **만드는 것보다 유지하는 게 비용**입니다. 새 워크플로를 추가할 때마다 대시보드도 고쳐야 하면, 2주 프로젝트가 대시보드 유지보수 프로젝트로 변질됩니다.
- 2026년 현재 이 규모에서 실제로 쓰이는 조합은 기성품입니다: **n8n 내장 Insights 대시보드**(워크플로 성공/실패/실행 이력), **LiteLLM 관리 UI**(모델별·작업별 비용 추적), **Uptime Kuma**(생존 감시 + 알림). 셋 다 설치만 하면 끝이고, 당신이 원한 "편리한 스킬 편집"은 n8n의 시각 편집기가 그 자체로 해결합니다(브라우저에서 드래그로 워크플로 수정 → 즉시 반영).

### 1-3. 클로드코드 세션 = 운영 런타임 구조의 문제

- 클로드코드 세션에서 돌던 프로젝트들은 **구독 토큰이 소진되면 멈춥니다.** "내가 토큰을 다 써도 자동화는 계속 돌아야 한다"는 당신의 핵심 요구와 구조적으로 양립 불가능합니다.
- 해법은 명확합니다: **운영은 API 키 기반**(Anthropic/OpenAI/Moonshot API + 로컬 모델)으로 돌리고, 클로드코드는 **이 시스템을 만드는 개발 도구로만** 씁니다. 구독 토큰과 운영이 완전히 분리됩니다.

### 1-4. 클로드 편향 검토

- Claude Agent SDK는 사실상 Anthropic 모델 전용이라 "다양한 AI 엔진 선택" 요구에 맞지 않습니다. → 채택 안 함.
- 오케스트레이션 코어는 특정 모델사에 종속되지 않는 **n8n + LiteLLM**으로 하고, Claude는 여러 엔진 중 하나(고품질 티어)로만 편성합니다.
- 참고로 요즘 유행하는 OpenClaw(구 Clawdbot/Moltbot)도 검토했습니다. 기능상으로는 당신 요구(슬랙 멀티에이전트, 멀티모델, 대시보드)와 정확히 겹치지만, 2026년 상반기 보안 사고 이력이 카테고리 최악입니다(원클릭 RCE CVE, 악성 스킬 824개 유통, 웹소켓 하이재킹 등). **매매 권한이 있는 시스템의 코어로는 부적격**이며, 굳이 쓴다면 "슬랙 대화 창구" 역할로만 격리해서 쓸 수 있습니다. 기본안에서는 제외합니다.

### 1-5. 살릴 것

- **하드웨어 3대 구성 자체는 우연히도 거의 이상적입니다.** 저전력 상시 서버(NAS) + 고성능 추론기(M3 Ultra 96GB) + 24/7 경량 워커(M4 24GB)는 딱 이 시스템에 필요한 3계층입니다. 문제는 하드웨어가 아니라 역할 배치였습니다.
- **슬랙도 살립니다.** 단, "에이전트가 사는 곳"이 아니라 "시스템이 나에게 보고하고 승인받는 곳"으로 역할을 바꿉니다. 채널 수도 줄입니다.

## 2. 목표 아키텍처

### 2-1. 설계 원칙 4가지

1. **결정적 실행, 선택적 지능**: 반복 업무의 뼈대는 n8n 워크플로(결정적)로 만들고, LLM은 워크플로 안의 "한 단계"(요약, 초안, 판단)로만 호출한다. LLM에게 프로세스 전체를 맡기지 않는다.
2. **단일 모델 관문**: 모든 LLM 호출은 LiteLLM 프록시 하나를 통과한다. 여기서 작업별 모델 라우팅, 비용 집계, 예산 초과 시 폴백(Claude → Kimi/OpenAI → 로컬)이 자동으로 일어난다.
3. **돈이 걸린 것은 격리**: 주식 매매는 LLM과 물리적으로 분리된 결정적 엔진이 실행하고, LLM은 읽기 전용 분석가로만 참여한다.
4. **만들지 말고 조립**: 대시보드, 스케줄러, 소셜 발행기 등은 전부 검증된 오픈소스를 조립한다. 직접 만드는 코드는 "우리 사업에만 있는 로직"(견적서 템플릿, 매매 전략 등)으로 한정한다.

**왜 이 배치인가**: 자동화의 심장(n8n)은 절대 꺼지지 않는 저전력 기기(NAS)에 있어야 합니다. 맥이 재부팅되거나 업데이트로 멈춰도 스케줄·워크플로·관제는 계속 돕니다. 반대로 96GB 모델 추론은 전기를 많이 먹으므로 Mac Studio는 "호출될 때만 일하는" 구조(LM Studio JIT 로드 + 유휴 시 자동 언로드)로 둡니다. Mac mini는 항상 켜둬도 부담 없는 24W급이므로 상시 소형 모델과 장중 내내 살아 있어야 하는 트레이딩 엔진을 맡습니다.

### 2-3. 모델 라우팅 정책 (토큰 최적화의 실체)

LiteLLM 설정 파일 하나로 구현되는 계층입니다:

| 티어 | 모델 | 담당 작업 | 비용 |
|---|---|---|---|
| 로컬 상시 | Qwen3.5-9B (mini) | 태깅, 분류, 리드 1차 스코어링, 짧은 요약 | 0원 |
| 로컬 고성능 | Qwen3.6-35B / gpt-oss-120b / EXAONE-32B (Studio) | 아카이브 요약, 퀴즈 생성, 블로그·포스팅 초안 | 0원 (전기값) |
| 클라우드 절약 | Kimi K2.x, GPT 계열 mini | 대량 배치 작업 중 품질이 필요한 것 | 저 |
| 클라우드 고품질 | Claude Sonnet/Opus (API), GPT 상위 | 최종 발행물 퇴고, 주식 분석 리포트, 견적서 문안 | 고 |

- **폴백 체인**: 고품질 티어 예산 소진 → LiteLLM이 자동으로 Kimi/로컬로 강등. 자동화는 멈추지 않고 품질만 한 단계 내려갑니다. "클로드 토큰 다 써도 계속 돌아야 한다"는 요구가 여기서 해결됩니다(운영은 애초에 구독 토큰을 쓰지 않음). **최종 폴백은 mini의 Qwen3.5-9B**입니다 — Studio는 JIT 로드 방식이라 콜드 스타트(수십 초)가 있으므로, 로컬 티어 호출에는 넉넉한 타임아웃을 설정하고 Studio는 절전 해제(caffeinate/전원 스케줄) 정책을 함께 둡니다.
- **비용 관제**: LiteLLM 관리 UI에서 워크플로별 가상 키를 발급해 "리드봇이 이번 달 얼마 썼는지"가 자동 집계됩니다.

### 2-4. 관제 시스템 (직접 제작 최소화)

| 무엇을 보나 | 도구 | 비고 |
|---|---|---|
| 워크플로 성공/실패/이력 | **n8n Insights** (내장) | 실패 시 슬랙 #ops 채널 자동 알림 |
| LLM 비용/사용량 | **LiteLLM 대시보드** (내장) | 예산·키별 집계 |
| 기기·서비스 생존 | **Uptime Kuma** (mini에 설치) | NAS 포함 전 기기 감시. 다운 시 슬랙/푸시 알림. 추가로 healthchecks.io 같은 외부 무료 하트비트 1개를 걸어 "mini 자체가 죽은 경우"까지 커버 |
| 트레이딩 상태 | 엔진이 슬랙 #trading에 정기 보고 | 포지션·손익·에러 |
| 통합 입구 | **Homepage** 류 정적 대시보드 1페이지 (mini) | 위 4개로 가는 링크 타일 모음 |

"편리한 스킬 편집" 요구 = **n8n 웹 편집기**가 담당합니다. 새 자동화를 추가하거나 프롬프트를 고치는 일이 코드 배포 없이 브라우저 안에서 끝나고, 버전 이력도 남습니다. 이것이 자체 대시보드를 폐기해도 되는 이유입니다.

### 2-5. 슬랙의 새 역할

채널을 "에이전트별"이 아니라 "기능별 소수"로 재편합니다:

- `#ops` — 시스템 알림(실패, 다운, 예산 경고)만
- `#approvals` — 사람 승인이 필요한 것(발행 전 포스팅 초안, 견적서 발송, 매매 전략 변경). n8n의 승인 대기 노드 + 슬랙 버튼으로 구현
- `#trading` — 트레이딩 엔진 정기 보고
- `#inbox` — 내가 시스템에 말을 거는 곳("이 주제 아카이브해줘", "견적서 만들어줘" → 슬랙 트리거 → n8n 워크플로 기동)

에이전트가 채널에 상주하며 대화하는 구조가 아니라, **워크플로가 필요할 때만 슬랙에 나타나는** 구조입니다. 토큰 소모가 급감하고, 프롬프트 인젝션 표면도 사라집니다.

### 2-6. 네트워크·보안 (인그레스) — 증권 계좌가 물린 시스템의 필수 조건

슬랙 트리거·승인 버튼은 외부(슬랙 서버)에서 우리 쪽으로 들어오는 요청이 필요합니다. 여기서 절대 하지 말아야 할 것이 **NAS 포트포워딩**입니다.

- **1순위**: 슬랙 연동은 가능하면 **Socket Mode**(아웃바운드 웹소켓만 사용, 인바운드 포트 개방 0)로 구성
- **인바운드 웹훅이 꼭 필요한 경우**(외부 서비스 웹훅 등): **Cloudflare Tunnel** 또는 **Tailscale Funnel**로만 노출 — 터널 밖에서 NAS/맥의 포트는 하나도 열리지 않게
- 관리 UI들(n8n, LiteLLM, Qdrant, 대시보드)은 **Tailscale 사설망 안에서만** 접근. 공인 인터넷에 관리 화면을 노출하지 않음
- **트레이딩 엔진은 LiteLLM 장애 시에도 살아야 하므로**, LLM 분석가 호출에 한해 LiteLLM 우회 직접 API 폴백 경로를 하나 둠 (엔진 자체는 LLM 없이도 동작하는 게 원칙)

## 3. 목표별 파이프라인 설계

### 3-1. 업무 관리 + 구글 일정 (1주차 필수)

- n8n의 Google Calendar/Gmail/Tasks 네이티브 노드로 구성. 매일 아침 브리핑(오늘 일정 + 미처리 업무 + 어제 자동화 결과 요약)을 슬랙 `#inbox`로 발송.
- `#inbox`에서 자연어로 지시("금요일에 A사 미팅 잡아줘") → n8n 슬랙 트리거 → LLM이 구조화(JSON) → Calendar 노드 실행. LLM은 파싱만, 실행은 노드가.
- 업무 저장소는 새로 만들지 말 것: 이미 쓰는 도구(Notion/Todoist/Google Tasks) 중 하나에 n8n을 연결하는 것으로 충분.

### 3-2. 사업 파트별 리드 발굴 → 블로그 발행 → 견적서 (1주차 필수)

사업 파트 대개편이 예정돼 있으므로, **파트별 하드코딩이 아니라 "파트 정의 테이블(키워드·타깃·템플릿)을 읽는 공용 파이프라인 1개"**로 설계합니다. 개편 후에도 테이블만 고치면 됩니다.

1. **수집**: 네이버 검색 API(공식, ~2.5만 콜/일) + 구글 검색/뉴스로 파트별 신호 수집 → mini의 Qwen3.5-9B가 1차 스코어링 → Qdrant/Postgres에 리드 저장
2. **콘텐츠**: 상위 리드/키워드 기반으로 Studio의 **Qwen3.6-35B 또는 A.X-4.0-Light(둘 다 Apache 2.0)**가 한국어 블로그 초안 생성 → (선택) Claude API로 퇴고 → `#approvals`에서 버튼 승인 → **WordPress REST API로 자동 발행** (※사업용 산출물이므로 비상업 라이선스인 EXAONE은 여기서 쓰지 않음 — §5-4)
3. **견적서**: 견적 요청(슬랙 지시 or 폼) → 품목·단가 테이블 조회 → HTML 템플릿 → PDF 변환(WeasyPrint/Gotenberg 컨테이너) → 승인 후 Gmail 발송

⚠️ **네이버 블로그·티스토리는 공식 글쓰기 API가 없습니다**(각각 2020/2024년 폐지). 셀레니움류 자동 발행은 계정 저품질/정지 리스크가 있는 그레이 마켓입니다. 완전 자동화는 자체 블로그(WordPress)에 걸고, 네이버 노출이 꼭 필요하면 "초안 자동 생성 → 내가 붙여넣기" 반자동으로 운영하세요.

### 3-3. 링크드인 자동 발행 (1주차 필수, 단 승인 게이트 있음)

- 공식 경로: LinkedIn "Share on LinkedIn"(w_member_social) 앱 심사 통과 필요 — **승인에 수주~수개월 걸릴 수 있으므로 1일차에 바로 신청**하세요. 개인 계정 1~2포스트/일 수준이면 충분한 API입니다.
- 승인 대기 동안의 브리지: Postiz(오픈소스, NAS Docker) 또는 Buffer 같은 기승인 SaaS로 발행. 파이프라인(초안 생성→승인→발행)은 동일하고 마지막 발행 노드만 교체하면 됩니다.
- 비공식 쿠키/헤드리스 자동화는 계정 제한 리스크 — 개인 브랜드 계정에는 권장하지 않음.

### 3-4. 관제 시스템 (1주차 필수)

§2-4 그대로. 1주차에는 n8n + LiteLLM + Uptime Kuma 3종 설치와 슬랙 알림 연결까지가 목표. 별도 개발 없음.

### 3-5. 주식 자동매매 — KR+미국 (2주차)

리서치 결론: **한국투자증권 KIS Developers OpenAPI를 기본으로 채택.**

- 판단 근거: 국내+미국(해외 5개 시장) 주문을 하나의 성숙한 REST/WebSocket API로 제공, 모의투자 지원, macOS/Linux 순수 REST(윈도우 OCX 불필요), 활발한 생태계(python-kis v2.1.6 유지보수 중, 공식 GitHub에 백테스터·MCP 포함).
- **토스증권 Open API는 2026년 6월 실제로 출시됐지만**(사전신청 순차 오픈) 아직 웹소켓 없음, 모의투자 없음, 토큰 1시간, 주문 경로 미검증 상태입니다. **지금은 기반으로 삼지 말고 사전신청만 해두고 2차 계좌/실험용으로 추적**하세요. (문서가 llms.txt/OpenAPI JSON 제공 등 AI 친화적이라 나중에 LLM 분석가의 읽기 전용 데이터원으로는 매력적입니다.)

구조는 반드시 분리합니다:

```
[LLM 분석가] (스케줄: 장전/장후)          [결정적 실행 엔진] (mini, 24/7)
 시황·뉴스·포트폴리오 분석                 전략 룰, 포지션 사이징, 하드 리스크 리밋
 → 구조화된 제안 JSON을 큐에 적재   ──→    제안을 가드레일로 검증 후에만 주문
 (읽기 전용, 주문 권한 없음)               주문 상태머신, 킬스위치, 슬랙 보고
```

- 엔진 필수 처리: 초당 20콜 레이트리밋(토큰 버킷), 웹소켓 자동 재접속, KRX/미국 정규·주간거래 세션 및 휴장일 캘린더, 미국은 지정가 주문만 지원됨을 전제.
- **토큰 갱신은 "자정 크론" 금지**: KIS 토큰은 24h 만료이고 재발급 시 기존 토큰이 무효화되는데, 자정(KST)은 미국 정규장 한복판입니다. 갱신은 고정 시각이 아니라 **만료 여유 시간 기반 + 세션 경계 앵커**(KR 개장 전 08:30, 미국 개장 전 22:00 KST 부근)로 스케줄하고, 갱신 직후 웹소켓 재구독까지 한 트랜잭션으로 처리하세요. 자정 부근 KIS 정기 점검 시간대도 피해야 합니다.
- 순서: 백테스트(공식 백테스터/backtrader + pykrx/FDR 과거데이터) → KIS 모의투자(모의계좌는 레이트리밋이 훨씬 낮음에 유의) → 소액 실전. **2주차에는 모의투자까지만 가는 것을 성공 기준으로 삼으세요.**
- 법적으로 본인 계좌 자동매매는 문제 없음(공식 API 약관 내). 타인 자금 운용·시그널 판매는 무등록 투자일임/자문업으로 불법 — 하지 마세요. 또한 초단타 주문/취소 반복 패턴은 시장질서 교란으로 계좌 플래그될 수 있습니다.

### 3-6. 관심 주제 학습 파이프라인 — 수집·퀴즈·아카이빙 (2주차)

1. RSS/뉴스레터/유튜브 자막을 n8n이 매일 수집 → 본문은 NAS에 원문 보관
2. Studio 로컬 모델이 요약 + 핵심 개념 추출 → BGE-M3 임베딩 → Qdrant 적재 (전 과정 0원)
3. 주 2~3회 gpt-oss-120b/Claude가 최근 아카이브에서 퀴즈 생성 → 슬랙 `#inbox`로 아침 퀴즈 발송, 버튼으로 답변 → 오답은 간격 반복(스페이스드 리피티션) 큐에 재적재
4. `#inbox`에서 질문하면 Qdrant RAG로 내 아카이브 기반 답변

### 3-7. 영상 콘텐츠 파이프라인 (2주차, E2E 1편이 목표)

키워드 발굴 → 스크립트 → 스토리보드 → 생성 → 더빙 → 발행:

- **키워드**: 유튜브 Data API `mostPopular` 차트(공식) + 네이버 데이터랩(공식) + Google Trends 공식 API(알파 대기신청, 폴백 pytrends/SerpApi)
- **스크립트/스토리보드**: Claude 또는 Kimi로 생성, `#approvals` 승인
- **이미지**: 로컬 FLUX(ComfyUI, M3 Ultra에서 실용적) 또는 GPT Image/Nano Banana API(텍스트 인 이미지·한글에 강함)
- **영상**: ⚠️ **로컬 영상 생성은 비실용**(Mac에서 2초 클립에 수십 분). 클라우드 API 사용 — 가성비 순: Wan 2.x($0.05/초) < Kling 3.0($0.10/초) < Veo 3.1 Fast($0.15/초). Sora API는 2026년 9월 중단 예정이라 제외.
- **더빙(한국어)**: Supertone 또는 Typecast API(한국어 운율이 ElevenLabs보다 우수), 대량·저가는 CLOVA Voice(₩4/1천자)
- **발행**: YouTube Data API — ⚠️ **미감사 앱은 업로드가 전부 '비공개' 처리**됩니다. 컴플라이언스 감사(2~4주)가 필요하므로 **이것도 1일차에 신청**. 쿼터는 일 6회 업로드 수준으로 개인에겐 충분.

### 3-8. 보이스 매니저 — 영어/일본어 학습 (2주차)

- 검증된 레퍼런스 스택: **pipecat + mlx-whisper(STT) + 로컬 LLM + Kokoro(영어 TTS)** — M시리즈에서 음성왕복 800ms 미만, 전부 로컬. Studio에서 구동(지연시간 때문에 mini 아님).
- 일본어 TTS는 Kokoro가 약하므로 **Style-Bert-VITS2 또는 VOICEVOX** 추가.
- ⚠️ Whisper large-v3-turbo를 쓰세요. 유행하는 Parakeet은 유럽어 전용이라 일본어/한국어 미지원.
- "내 표현 기록": 대화 로그를 파이프라인이 자동으로 문장 단위 추출 → 교정·대안 표현 생성 → 학습 아카이브(Qdrant)에 적재 → §3-6의 퀴즈 루프에 합류. 어학과 지식 학습이 같은 인프라를 공유합니다.

## 5. 리스크와 금지 사항

1. **LLM에게 주문 권한을 직접 주지 말 것.** 분석가(읽기)와 집행자(결정적 코드)의 분리는 이 설계에서 타협 불가 항목입니다.
2. **네이버 블로그 셀레니움 자동 발행 금지** — 개인 브랜드 계정의 저품질/정지 리스크가 이득보다 큽니다.
3. **관제 대시보드를 다시 만들고 싶은 유혹을 참을 것.** 기성 도구 3개 조합이 부족하다고 느껴지는 시점은 시스템이 실제로 한 달 이상 돌아간 후여야 합니다.
4. **EXAONE 라이선스 주의**: 비상업 라이선스이므로 개인 학습·블로그엔 무방하나, 사업 수익에 직결되는 산출물(리드 스코어링, 견적 문안)은 Apache 2.0인 Qwen 계열/A.X-4.0-Light 또는 클라우드 모델로 처리하세요.
5. **백업은 스냅샷만으로 부족**: 실행 중인 Postgres의 파일시스템 스냅샷은 정합성이 보장되지 않습니다. **매일 밤 `pg_dump` 잡** + n8n 워크플로 export + **`N8N_ENCRYPTION_KEY`를 반드시 별도 위치에 백업**(이 키가 없으면 복원해도 크리덴셜 전부 복구 불가). 2주차 마지막 날 복원 리허설을 한 번 하세요.
6. **컨테이너 버전 고정**: n8n은 릴리즈가 빠릅니다. `latest` 태그 금지, compose에 정확한 버전 태그를 박고 업그레이드는 의도적으로만.
7. **장중 mini 자원 격리**: 트레이딩 엔진과 로컬 모델이 같은 mini에 삽니다. Ollama `keep_alive`·메모리 상한을 걸고, 대량 임베딩 배치는 장 시간(KRX 09:00~15:30, 미국 22:30~05:00 KST)을 피해 스케줄하세요.
8. **시크릿 관리**: API 키·계좌 정보는 n8n 크리덴셜 저장소와 .env로만 관리하고, 외부 공유 폴더·공인망 노출 경로에 절대 두지 마세요. 관리 UI는 Tailscale 사설망 전용(§2-6).
9. **OpenClaw류 도입 시**: 반드시 루프백 바인딩 + 서드파티 스킬 금지 + 매매·발행 권한 미부여 조건으로만. 기본안에서는 불필요합니다.
10. **수치는 리서치 시점 기준**: 본문 중 영상 API 단가, 음성 왕복 지연(<800ms는 최적 조건 기준), TTS 품질 비교, Sora 중단 일정 등은 2026년 7월 리서치 기준 추정치입니다. 결제·의존 결정 전에 해당 시점 가격/정책을 재확인하세요.

## 6. 이 설계가 당신의 요구를 어떻게 충족하는지 (체크리스트)

| 요구 | 해결 방식 |
|---|---|
| 프로젝트 성격별 업무 자동화 | n8n 워크플로 = 프로젝트별 1개, 공용 패턴(수집→LLM 단계→승인→실행) 재사용 |
| 업무 관리 + 구글 일정 | n8n Google Calendar/Tasks 노드 + 슬랙 아침 브리핑·자연어 등록 (§3-1) |
| 최신 자료 수집·퀴즈·아카이빙 | 수집→로컬 요약→Qdrant 아카이브→아침 퀴즈+간격 반복 (§3-6) |
| 리드 발굴·블로그·견적서 | 파트 정의 테이블 기반 공용 파이프라인 + WordPress 발행 + PDF 견적 (§3-2) |
| 링크드인 자동 발행 | 공식 API 심사(0일차 신청) + 대기 중 Postiz 브리지 (§3-3) |
| 작업별 AI 엔진 선택 (탈클로드) | LiteLLM 라우팅: 로컬 2티어 + Kimi/OpenAI + Claude, 설정 파일로 교체 자유 |
| 영상 콘텐츠 생산 | §3-7 파이프라인 (로컬 이미지 + 클라우드 영상 API + 한국어 TTS) |
| 효율적 프로젝트 관리·관제 | n8n Insights + LiteLLM UI + Uptime Kuma(mini) + 슬랙 알림 (제작 0) |
| 스킬 편집 편의 | n8n 브라우저 편집기 (배포 없이 수정) |
| 토큰 소진에도 무중단 | 운영은 API 키 기반 + 예산 소진 시 자동 폴백 체인 → 로컬 모델 |
| 어학 보이스 매니저 | 전량 로컬 음성 스택 + 표현 아카이빙이 학습 퀴즈 루프와 통합 |
| 주식 자동매매 (KR+US) | KIS OpenAPI + 분리형 엔진/분석가, 모의투자부터 |
