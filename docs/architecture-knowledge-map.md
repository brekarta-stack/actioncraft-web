# 아키텍처 설계 지식 지도 — 당신 시스템 기준 전문가 정리

> 대상 시스템: 개인 에이전트 오케스트레이션(n8n 워크플로 + LiteLLM 게이트 + 로컬/클라우드 멀티모델 + 주식 매매 + 콘텐츠 발행 + 관제), 하드웨어 NAS/Mac Studio/Mac mini, 1인·2주.
> 각 항목에 **[필수]/[권장]/[과잉]** 등급을 붙였다(2주·초보 기준 우선순위).

---

## 0. 개념 계층 — 라우팅은 어디에 있나

아키텍처 설계 = "무엇을, 어디에, 어떻게 배치하고 연결하고 지키는가"의 전체 결정. 그 아래에 대략 이런 하위 영역이 있다:

```
아키텍처 설계 (전체)
├─ 1. 컴포넌트·토폴로지 설계   (무슨 서비스를 어느 기계에)
├─ 2. 데이터 아키텍처          (저장·상태·스키마·데이터 흐름)
├─ 3. 통합 아키텍처           (API·MCP·인증·시크릿)
├─ 4. 모델/추론 아키텍처  ←── ★ 라우팅 규칙이 여기 속함
├─ 5. 오케스트레이션/워크플로 설계 (결정적 vs 에이전트, 이벤트 vs 스케줄)
├─ 6. 신뢰성·가용성 설계       (장애점·페일오버·헬스체크)
├─ 7. 보안 아키텍처           (네트워크·최소권한·인젝션·매매 격리)
├─ 8. 관측성(Observability)   (로그·메트릭·트레이스·알림)
├─ 9. 배포·운영(Ops)          (컨테이너·버전·백업·복구·IaC)
├─ 10. 비용 아키텍처(FinOps)   (예산·토큰 경제)
└─ 11. 진화·이주 설계          (확장성·마이그레이션·버전)
```

라우팅은 이 중 4번의 세부다. 즉 "좋은 라우팅"만으로 좋은 아키텍처가 되지 않는다 — 나머지 10개가 함께 서야 한다.

## 1. 컴포넌트·토폴로지 설계 [필수]

**정의**: 어떤 독립 컴포넌트(서비스)로 쪼개고, 각각을 어느 하드웨어에 두고, 어떤 경계로 나눌지.

**핵심 개념**
- **관심사 분리(Separation of Concerns)**: 하나의 서비스는 하나의 책임. 매매 엔진, 발행 워크플로, 모델 게이트는 각각 독립 컨테이너.
- **결합도/응집도(Coupling/Cohesion)**: 서로 덜 얽히게(느슨한 결합), 관련된 건 뭉치게(높은 응집).
- **상태ful vs 상태less**: 매매 엔진·DB는 상태를 가짐(신중히 배치), n8n 워크플로 실행은 가능하면 무상태로.

**당신 시스템의 결정**: NAS=24/7 백본, Studio=온디맨드 추론, mini=상시 경량+관제. 경계 원칙 = "돈 걸린 것(매매)은 격리, 상시 필요한 것은 저전력 기기에".

---

## 2. 데이터 아키텍처 [필수]

**정의**: 데이터가 어디 저장되고, 무엇이 "진실의 원천(Source of Truth)"이며, 어떻게 흐르고 백업되는가.

**핵심 개념**
- **단일 진실 원천(SSOT)**: 리드·업무·매매 상태 각각의 "정본"이 어디인지 하나로 못박기. 두 곳에 있으면 반드시 불일치가 난다.
- **정형 vs 벡터 저장**: 관계형(Postgres) = 구조화 데이터/상태, 벡터DB(Qdrant) = 임베딩/RAG 검색. 역할이 다르다.
- **RAG(검색 증강 생성)**: 문서를 임베딩→벡터DB 저장→질문 시 유사 문서 검색→LLM에 컨텍스트로 주입. 학습 아카이브·어학 표현이 이 파이프라인.
- **스키마 마이그레이션**: DB 구조 변경을 버전 관리(사업 파트 대개편 시 필수).
- **백업 정합성**: 실행 중 DB의 파일 스냅샷은 깨질 수 있음 → `pg_dump` 같은 논리 백업. **RPO(얼마만큼의 데이터 손실 허용)/RTO(얼마 만에 복구)** 개념으로 백업 주기 결정.

**당신 시스템의 함정**: n8n 크리덴셜은 `N8N_ENCRYPTION_KEY`로 암호화됨 → 이 키를 잃으면 복원해도 무용. 별도 백업 필수.

## 3. 통합 아키텍처 [필수]

**정의**: 외부 서비스(증권사·구글·슬랙·소셜)와 어떻게 연결하고 인증·시크릿을 관리하는가.

**핵심 개념**
- **API 유형**: REST(요청-응답), WebSocket(실시간 스트리밍, 매매 체결가), Webhook(상대가 나를 호출).
- **MCP(Model Context Protocol)**: LLM에게 도구를 표준 방식으로 물려주는 규격. 커넥터를 붙일 때의 공통 언어.
- **인증**: OAuth2(구글·링크드인), API Key(모델·KIS), 토큰 만료·갱신(KIS 24h, 토스 1h).
- **시크릿 관리**: 키를 코드에 하드코딩 금지. `.env`/n8n 크리덴셜/시크릿 매니저로 분리, 로테이션(주기적 교체), 백업에 포함하되 암호화.
- **레이트리밋 대응**: 상대 API의 초당 호출 제한(KIS 20/s)을 토큰 버킷으로 준수.
- **멱등성 키(Idempotency Key)**: 같은 요청이 두 번 가도 한 번만 처리되게. ★매매·발행에 치명적(§중요).

## 4. 모델/추론 아키텍처 [필수] ← 라우팅이 여기

**정의**: 어떤 모델을, 어디서 서빙하고, 작업마다 어떻게 고르고, 실패 시 어떻게 대체하고, 비용을 어떻게 통제하는가.

**핵심 개념**
- **모델 게이트웨이**: 모든 호출을 한 관문(LiteLLM)으로. 추상화·집계·교체 용이.
- **라우팅**: 작업→모델 매핑(역할 별칭). 규칙 기반 vs 에이전트 판단.
- **폴백 체인 & 강등(Graceful Degradation)**: 1순위 실패/예산초과 시 자동으로 다음 티어로. "멈춤" 대신 "품질 한 단계 하락".
- **서빙 스택**: 로컬(LM Studio 헤드리스/Ollama-MLX)의 JIT 로드·유휴 언로드, 콜드스타트 지연 대비 타임아웃.
- **용량 계획(Capacity Planning)**: Q4≈0.5GB/1B 공식으로 메모리 예산. 96GB에 어떤 조합이 공존 가능한지.
- **구조화 출력(Structured Output)**: LLM이 자유 텍스트 대신 검증 가능한 JSON을 내게 강제(매매 제안·라우팅 판단).
- **평가(Eval)**: 로컬 모델 산출물 품질을 자동 검증(테스트·스키마·LLM-as-judge)해 자동 승급 트리거.

**당신 시스템**: routing-design.md의 별칭·폴백·4대 게이트가 이 영역의 산출물.

## 5. 오케스트레이션 / 워크플로 설계 [필수]

**정의**: 여러 단계·에이전트를 언제·어떤 순서로·어떤 트리거로 실행하는가.

**핵심 개념**
- **결정적 워크플로 vs 자율 에이전트**: 반복 업무는 결정적(n8n), 판단이 열린 문제만 에이전트. 이 시스템의 핵심 철학.
- **트리거 유형**: 스케줄(cron), 이벤트/웹훅, 수동(슬랙 지시). 폴링 vs 이벤트(가능하면 이벤트).
- **멱등성 & 정확히 한 번(Exactly-once)**: 재시도해도 부작용 1회. 매매 주문·게시·이메일에 필수.
- **재시도·백오프·서킷브레이커**: 실패 시 지수 backoff 재시도, 계속 실패하면 회로 차단(무한 재시도 방지).
- **데드레터 큐(DLQ)**: 끝내 실패한 작업을 버리지 말고 격리해 사람에게.
- **휴먼인더루프(HITL)**: 발행·매매전략 변경·견적 발송 전 사람 승인 게이트(#approvals).
- **사가/보상 트랜잭션**: 여러 단계 중 일부 실패 시 롤백/보상 로직(예: 게시했는데 후속 실패 시 정정).

## 6. 신뢰성 · 가용성 설계 [필수]

**정의**: 무언가 죽어도 시스템이 버티고 알아채고 복구하는가.

**핵심 개념**
- **단일 장애점(SPOF)**: 하나 죽으면 전체가 멈추는 지점. NAS가 SPOF인지, 그렇다면 감내할지.
- **감시자-감시대상 분리**: 관제(Uptime Kuma)를 감시 대상(NAS) 밖(mini)에, mini는 외부 하트비트가 감시. 사각지대 제거.
- **자동 복구**: `restart: unless-stopped`, 재부팅 후 자가 기동.
- **우아한 성능저하(Graceful Degradation)**: 일부 기능이 죽어도 나머지는 계속(모델 폴백이 이 개념의 한 예).
- **가용성 목표**: 개인 시스템은 99% 정도로 충분. 과잉 이중화는 [과잉].

## 7. 보안 아키텍처 [필수 — 증권 계좌 물림]

**정의**: 자산·계정·데이터를 어떻게 지키는가.

**핵심 개념**
- **최소 권한(Least Privilege)**: 각 서비스·키에 딱 필요한 권한만. 매매 분석 LLM은 읽기 전용 키.
- **네트워크 분리·제로트러스트**: 관리 UI는 사설망(Tailscale)에만. 인바운드 포트 최소화(슬랙 Socket Mode). 포트포워딩 금지.
- **시크릿·자격증명 위생**: 키 로테이션, 저장소 암호화, 로그에 키 노출 금지.
- **프롬프트 인젝션 / 에이전트 보안**: 외부 콘텐츠(웹·이메일)가 LLM을 조종하는 공격. 도구 권한 최소화, 신뢰 못 할 입력과 특권 도구 분리, 서드파티 "스킬" 무분별 설치 금지.
- **매매 격리**: LLM은 제안만, 실행은 결정적 엔진이 가드레일 검증 후. 킬스위치·일손실한도.
- **공급망 보안**: 컨테이너 이미지·플러그인 출처 신뢰·버전 고정.

---

## 8. 관측성(Observability) [필수]

**정의**: 지금 뭐가 돌고, 뭐가 실패했고, 왜 그런지 알 수 있는가.

**핵심 개념**
- **3대 축**: 로그(무슨 일이), 메트릭(수치·추세), 트레이스(요청의 여정).
- **LLM 특화 관측**: 토큰·비용·지연·품질(eval). LiteLLM UI가 비용·키별 집계.
- **알림(Alerting)**: 실패·다운·예산 초과를 슬랙으로. **알림 피로** 방지(중요한 것만).
- **관제는 조립**: n8n Insights + LiteLLM UI + Uptime Kuma. 자체 대시보드 재제작은 [과잉].

## 9. 배포 · 운영(Ops) [필수]

**정의**: 어떻게 올리고, 버전을 관리하고, 재현 가능하게 만드는가.

**핵심 개념**
- **컨테이너·오케스트레이션**: Docker Compose(1인 규모 적합). Kubernetes는 [과잉].
- **코드형 인프라(IaC) / 선언적 설정**: docker-compose.yml, LiteLLM config.yaml, n8n export를 Git 버전관리 → 재현·롤백 가능.
- **이미지 버전 고정(Pinning)**: `latest` 금지, 정확한 태그. 의도적 업그레이드.
- **환경 분리**: 최소한 매매는 모의(paper)↔실전 분리. 개발/운영 구분.
- **백업·재해복구(DR)**: 정기 논리 백업 + 오프사이트(B2) + 복원 리허설.
- **롤백 전략**: 잘못된 변경을 되돌리는 경로 사전 확보.

## 10. 비용 아키텍처(FinOps) [권장]

**정의**: 돈(특히 토큰)을 어떻게 통제·최적화하는가.

**핵심 개념**
- **토큰 경제학**: 입력/출력 단가, 컨텍스트 길이가 비용. 로컬로 오프로드해 절감.
- **예산 캡·쿼터**: 별칭/워크플로별 월 상한, 80% 경고/100% 강등.
- **비용 귀속(Attribution)**: 가상 키로 "어느 워크플로가 얼마 썼는지" 집계.
- **캐싱**: 반복 프롬프트·임베딩 재사용으로 비용 절감.

---

## 11. 진화 · 이주 설계 [필수 — 기존 40잡 존재]

**정의**: 기존 시스템에서 새 시스템으로 어떻게 안전하게 옮기고, 앞으로 어떻게 확장하는가.

**핵심 개념**
- **교살자 패턴(Strangler Fig)**: 낡은 시스템을 한 번에 갈아엎지 않고, 새것이 기능을 하나씩 흡수→검증→옛것 제거. 당신의 celery/slackbot 40잡 이주 전략.
- **"켜기 전에 끈다"**: 신·구 동시 가동으로 인한 중복 실행(중복 주문·이중 게시) 방지.
- **확장성 vs 진화성**: 개인 시스템에 수평 확장(스케일아웃)은 대개 [과잉]. 대신 **진화성**(새 워크플로·모델을 싸게 추가) 중시 → 별칭 추상화·설정형 파트 테이블.
- **결정 기록(ADR)**: 중요한 설계 결정을 근거와 함께 문서화(왜 KIS인지, 왜 n8n인지). 미래의 나·클로드코드가 참조.

---

## 12. 비기능 요구(-ilities) 요약

아키텍처 "품질"은 기능이 아니라 다음 속성들로 평가된다. 당신 시스템의 우선순위:
1. **신뢰성(Reliability)** — 무중단 요구. 최우선.
2. **보안(Security)** — 증권 계좌. 최우선.
3. **비용효율(Cost)** — 토큰 최적화. 높음.
4. **유지보수성(Maintainability)** — 1인 운영. 높음(단순함이 미덕).
5. **관측성(Observability)** — 관제 요구. 높음.
6. **진화성(Evolvability)** — 사업 대개편. 높음.
7. 확장성(Scalability) — 1인이라 낮음. 과투자 금지.

## 13. 당신 시스템에서 특히 치명적인 5가지 (놓치면 사고)

1. **멱등성/정확히 한 번** — 재시도가 중복 주문·이중 게시를 내면 돈·평판 사고. 모든 부작용 있는 액션에 멱등성 키.
2. **시크릿·매매 격리** — 키 유출·LLM 주문권한은 재앙. 최소권한+격리.
3. **단일 장애점 & 자기맹목 관제** — NAS와 함께 관제가 죽으면 소리 없이 전체 정지. 감시 분리.
4. **프롬프트 인젝션** — 수집한 외부 콘텐츠가 발행/매매 워크플로를 조종. 신뢰경계 설정.
5. **백업 정합성 & 암호화 키** — 스냅샷만 믿다 복원 실패. 논리 백업+키 별도 보관+복원 리허설.
