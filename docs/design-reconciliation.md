## 1. 일치점 (양쪽 수렴 — 교차검증 완료, 이견 없음)
- n8n(NAS) 결정적 백본 + LLM은 "한 스텝"으로만. 채널별 상시 챗 에이전트 폐기.
- LiteLLM 단일 관문 + 역할 별칭 + 예산캡(80%경고/100%강등) + 폴백체인. 무중단 = 자동 강등.
- 운영을 Claude Code 구독토큰에서 분리(API/로컬). Claude Code는 개발도구로 강등.
- 매매: LLM 분석가(읽기전용, 주문권한 0) ↔ 결정적 집행엔진 격리. KIS OpenAPI(국내+미국, 모의→실전), 토큰 세션앵커 갱신, 레이트리밋 토큰버킷, 웹소켓 재접속.
- 관제 3종(n8n Insights + LiteLLM UI + **Uptime Kuma@mini**) + Slack #ops. 자체 대시보드 재제작 금지.
- Slack Socket Mode(인바운드 0) 4채널, 포트포워딩 금지, Tailscale 사설망.
- 백업: pg_dump + N8N_ENCRYPTION_KEY 별도보관 + 복원 리허설, 이미지 버전핀.
- 교살자 이주 + "켜기 전에 끈다"(중복 방지). LinkedIn/YouTube 0일차 신청. 블로그=WordPress(네이버 반자동).
- 한국어=Qwen/A.X(Apache2.0), EXAONE=개인학습 전용(비상업). 매매 시그널 판매 불법.

## 3. 빈틈 (한쪽에만 있음 — 통합 시 흡수)

### 3-A. 내 세션에만 있음 → 사용자 문서에 흡수 필요
- **실제 사업 3개 확정**: 페이퍼크래프트(biz-a)·지역재생(regen/biz-b)·스타트업강의(edu/biz-c). 문서는 아직 "파트 대개편 예정"으로만 표기 → biz-b/c 플레이스홀더 확정.
- **24채널 재편표**(KEEP/MERGE/PRUNE) — 문서의 "40잡 이주 순서 미정"을 구체화.
- **프로덕트 개발 관리**(운영 vs 개발 분리, projects.yaml + 주간 다이제스트) — 문서에 아예 없는 영역(impact-ledger·proof-market 등).
- **도메인 게이트 재정렬**(Track A 인프라 먼저 / 도메인종속 뒤로) — 문서 G는 원래 순서(리드·블로그 1주차).
- **스킬 실측**: agent-config/skills 비어있음 + "효과 검증 후 이식"(qc-eval 로그 활용). 문서는 스킬 이식을 당연시.
- **KIS 이미 부분통합**: kis-balance 읽기전용 잔고조회 가동 중 → 매매 착수 기점.

### 3-B. 사용자 문서에만 있음 → 내 통합안에 흡수 필요
- **게이팅 3결정**(매매위치·멱등성·RTO/RPO) — 명시적 착수 전 관문. 내 세션 미다룸 → §다음 단계에서 확정.
- **라우팅 별칭 11종 + 4대 게이트**(자동승급·예산·프라이버시·시간대) — 내 "spec/codegen/review"보다 정교. 채택.
- **IP 안정화**(Tailscale MagicDNS·DHCP 예약·컨테이너명 통신·NAS 랜선 상시) — 내가 안 다룬 실운영 함정. 채택.
- **11영역 지식맵 + 성숙도 점검표** — 설계 누락 방지 프레임. 채택.
- **모델 로스터 구체**(Coder-30B 상시/Next-80B JIT/9B mini) + JIT 로드 정책 — 채택.
- **안티패턴 목록**(OpenClaw 보안사고 격리, 과분할 금지 등) — 채택.

## 5. 사용자 결정 반영 (2026-07-22): **미니 제거 → 2기기 아키텍처**
사용자 지시: "디바이스 관리 대수 축소 — 미니 삭제 희망." RAM: **32GB 확정.**
검토 결과 **미니 제거는 가능**하며, 상충 C1·C2·C4가 한꺼번에 해소된다. 미니의 역할을 3곳으로 재배치:

| 미니가 하던 것 | 이전 위치 | 근거 |
|---|---|---|
| 매매 엔진(python-kis) | **→ NAS 컨테이너** | 결정적 코드(추론 아님)라 NAS CPU로 충분. NAS가 진짜 24/7 티어 (C1 판정) |
| voicebridge 음성비서 | **→ Studio** | 오디오 하드웨어 필요 → 데스크톱 맥. 대화형이라 온디맨드 허용 (C4 판정) |
| Uptime Kuma(감시자) | **→ NAS(컨테이너 감시) + 외부 클라우드 모니터(NAS 감시)** | "NAS 자체가 살아있나"는 healthchecks.io/UptimeRobot이 감시 → 감시자·감시대상 분리 유지(오히려 로컬 정전에 강함) |
| 상시 소형 Qwen9B/임베딩 | **→ Studio(온디맨드) + 폴백=클라우드 저가** | 24/7 로컬 티어 상실 → Kimi로 대체(월 수 달러). BGE-M3 임베딩은 NAS CPU or Studio |

**결과 = 2기기**: NAS(24/7 백본: n8n·Postgres·LiteLLM·매매·Kuma·임베딩) + Studio(온디맨드 헤비: 로컬추론·음성·이미지·개발) + 외부 클라우드 모니터.
**불변 원칙 위반 없음**: 무중단(운영·매매 NAS 24/7), 감시분리(외부 클라우드), 멀티모델 — 전부 유지.
**치르는 비용(정직)**: ① 24/7 로컬 저가 티어 상실 → 클라우드 저가로 대체(비용 미미) ② 음성은 Studio가 깨어 있어야(온디맨드 웨이크 수초 or Studio 상시화) ③ 물리적 음성 위치가 Studio 있는 곳으로 이동.
**제거 시점**: 40잡 NAS 이관 + voicebridge Studio 이식 검증 후. **대체 검증 전 미니 삭제 금지**(2주 병행).

→ 판정 갱신: **C1=NAS · C2=소멸(미니 없음) · C4=Studio로 이식 · C3=pgvector · C5=32GB · C6=사용자 확정 수용.**
