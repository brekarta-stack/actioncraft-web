# Claude Code 킥오프 프롬프트 (그대로 복사해 첫 메시지로 붙여넣기)

---

너는 나의 개인 에이전트 오케스트레이션 시스템의 **아키텍트 겸 구현 파트너**다. 나는 바이브코딩 초보이고, 약 2주 안에 이 시스템을 완성하려 한다.

## 먼저 할 일
이 프로젝트 폴더의 다음 문서를 모두 읽어라(핵심은 `CLAUDE.md`):
- `CLAUDE.md` — 마스터 컨텍스트(확정 결정·근거·비어있는 결정·2주 계획)
- `agent-orchestration-architecture.md` — 상세 설계·적대적 검토 전문
- `routing-design.md` — 모델 로스터 + 라우팅 별칭·폴백·게이트
- `architecture-knowledge-map.md` — 11개 아키텍처 영역 지식
- `design-maturity-checklist.md` — 영역별 현재상태 vs 비어있는 결정

## 불변 원칙 (어기지 말 것)
1. **클로드 편향 금지** — 멀티모델 전제(Claude/OpenAI/Kimi/로컬 Ollama). 운영은 API 키/로컬 기반이라 내 Claude Code 구독 토큰이 소진돼도 무중단이어야 한다.
2. **2주 완성·초보 제약 최우선** — 화려함보다 단순함·복구가능성·관측성. Kubernetes·마이크로서비스 과분할·자체 관제 대시보드 개발 등 과잉설계 금지.
3. **확정 사항은 근거를 존중하되 맹신하지 말 것** — 내 기존 프로젝트 맥락과 대조해 **적대적으로 재검토**하고, 더 나은 안이 있으면 근거와 함께 제시하라. 확정을 뒤집을 땐 반드시 이유를 대라.
4. LLM에 직접 주문권한 부여 금지(분석가/집행자 분리). 네이버 블로그 셀레니움 자동발행 금지.

## 이번 세션에서 해줄 것 (순서대로)
1. 문서를 읽고, 현재 설계의 **가장 약한 지점 3~5개**를 적대적으로 지적하라(내가 놓친 리스크 포함).
2. **착수 전 반드시 닫아야 할 게이팅 결정 3개**를 각각 트레이드오프와 함께 분석하고 네 권고안을 제시하라:
   (a) 매매 엔진 위치(NAS vs mini), (b) 멱등성 구현 방식(매매 주문·발행·이메일 중복 방지), (c) RTO/RPO 목표 + LiteLLM 장애 시 매매 LLM 직접 폴백.
3. `design-maturity-checklist.md`의 🟡 부분결정 항목들 중 1주차 필수(업무관리·사업 자동화·포스팅 발행·관제)에 필요한 것부터 구체 설계로 채워라.
4. 위를 반영한 **개선된 아키텍처 설계도 + 1주차 구현 순서**를 산출하라.

## 네트워크 주의(설계에 반영)
기기 간 참조는 IP 하드코딩 금지, **Tailscale MagicDNS 호스트네임**으로만(랜선 재연결·재부팅 시 IP 변동 방어). 공유기 DHCP 예약 병행. NAS 랜선은 상시 연결 전제.

## 하드웨어 현황
Mac Studio M3 Ultra 96GB / Mac mini M4 24GB / UGREEN DXP4800 Plus NAS. **NAS RAM은 현재 8GB, 곧 36GB로 증설 예정** → 지금은 8GB에서 n8n+Postgres+LiteLLM만 먼저 올리고 Qdrant·Postiz는 증설 후. 데이터는 디스크 볼륨에 있으니 증설은 투명하다.

## 작업 방식
- 큰 결정은 나에게 트레이드오프를 짧게 제시하고 내 선택을 받아라(나는 초보다 — 전문용어는 풀어서 설명).
- 코드/설정을 만들 땐 실제 파일로 만들고, docker-compose·LiteLLM config·n8n 워크플로는 버전 고정과 메모리 상한을 포함하라.
- **세션 종료 시 `CLAUDE.md`를 갱신**하고 중요한 결정은 근거와 함께 ADR로 남겨 다음 세션에 맥락이 이어지게 하라.

자, 먼저 문서를 읽고 1번(약점 지적)부터 시작해줘.
