## 0. 운영 목적 (사용자 확정, 2026-07-22)

1. 프로젝트 성격별 업무 자동화: 주식 자동매매 / 관심주제 수집→학습(퀴즈·아카이빙) / 링크드인 자동 포스팅 / 업무·구글 일정 관리 / 사업파트별 리드 발굴→블로그 포스팅→견적서 자동 발행
2. 작업별 맞춤 AI 엔진 선택 (Claude·OpenAI·Kimi·ollama 등 멀티엔진)
3. 영상 콘텐츠 생산 파이프라인 (키워드→스크립트→스토리보드→이미지/영상 생성→더빙→퍼블리싱)
4. 효율적 프로젝트 관리 + 관제 (슬랙이 부족하면 대체 가능)
5. 토큰 최적화 + 끊김 없는 업무 (Claude Code 토큰 소진돼도 자동화·관제는 계속)
6. 보이스 매니저 어학 학습 (영어/일본어, 내 표현 기록)

## 1. 핵심 판정 3가지 (적대적 검토 결과)

### 판정 1 — 자동화의 몸통은 "Claude Code 세션"이 아니라 "24h 서비스"여야 한다
목적 5(토큰 소진돼도 계속)가 결정적이다. **Claude Code = 공장을 짓는 도구, n8n/서비스 = 공장.**
자동화 런타임이 Claude 구독 토큰에 묶이면 목적 5는 구조적으로 불가능하다.
→ 자동화 본체는 NAS의 상시 서비스로, LLM 호출은 API 종량제(멀티엔진 라우터)로 분리한다.

### 판정 2 — 스튜디오는 "필수"가 아니라 "추론 어플라이언스"다
7가지 목적 중 **스튜디오가 없으면 불가능한 것은 하나도 없다.** 전부 API로 대체 가능.
스튜디오의 실제 가치는 3개뿐: ① 토큰 소진/장애 시 로컬 폴백(T1 실측 83 tok/s)
② 상시·대량 호출의 API비 절감 ③ 로컬 이미지 생성·무거운 실험.
→ 스튜디오를 **시스템의 기둥에서 뺀다.** 있으면 라우터의 한 티어로 꽂히고, 꺼져 있어도
시스템은 완결된다. (팔아도 시스템은 돌아간다 — 이미 보유했으니 어플라이언스로 활용)

### 판정 3 — 전면 재구축도, 전면 보존도 아니다 (교살자 무화과 전략)
- **신설**: 백본(라우터·워크플로·관제·DB)은 NAS에 새로 짓는다 — 기존에 없던 층이라 충돌 없음.
- **보존**: 검증된 자산 — voicebridge(음성비서), google MCP, restic 백업, Slack 채널 체계, 미니의 40개 잡(대체 검증 전까지 계속 가동).
- **단계 은퇴**: papercraft launchd 함대·celery·slackbot은 n8n 플로우가 하나씩 대체 검증될 때마다 하나씩 끈다. 빅뱅 컷오버 금지.

## 2. 타깃 아키텍처

```
┌─ 인터페이스 ────────────────────────────────────────────────┐
│ Slack(명령·알림·승인버튼)  ·  대시보드(관제 웹)  ·  보이스(미니)   │
│ 폰/노트북 어디서든: Tailscale + Claude Code Remote Control      │
└──────────────────────────┬──────────────────────────────────┘
┌─ NAS DXP4800+ (24h 백본 — 유일한 상시 필수 기기) ─ Docker ──┐
│ • LiteLLM 라우터: 작업별 엔진 매핑 + 폴백체인 + 예산/사용량 집계  │
│     claude-api / openai / kimi / ollama@studio / ollama@mini   │
│ • n8n: 모든 자동화 파이프라인 (수집·퀴즈·SNS·리드·견적·영상·매매) │
│ • Postgres(상태·아카이브·어학표현) + Redis(큐)                  │
│ • Uptime Kuma: 전 기기·전 서비스 워치독 → Slack (관제의 닻)      │
│ • 저장: 자료 아카이브 · 모델 가중치 · restic 백업 타깃            │
└──────────────────────────┬──────────────────────────────────┘
┌─ Mac Mini 24GB (24h, macOS 접점) ──┐ ┌─ Mac Studio 96GB (선택 티어) ─┐
│ • voicebridge 음성비서(마이크·스피커) │ │ • ollama/MLX: qwen3.6-35b 서빙 │
│ • 어학 학습 보이스(표현→NAS DB 기록)  │ │   (라우터의 local-large 티어)    │
│ • Claude Code headless(무거운 판단  │ │ • 로컬 이미지 생성(Flux/DrawT)   │
│   스텝을 n8n이 CLI로 호출)          │ │ • 무거운 개발 세션·실험           │
│ • macOS 전용 잡(AppleScript류)     │ │ • 꺼져 있어도 시스템 무영향        │
│ • 기존 40잡: 대체될 때까지 계속 가동  │ │   (라우터가 API로 자동 폴백)      │
└────────────────────────────────────┘ └───────────────────────────────┘
```

**한 줄**: NAS가 백본(자동화·라우팅·관제·저장), 미니가 macOS 접점(음성·기존 함대), 스튜디오는 꽂으면 좋은 추론 어플라이언스. **어떤 맥이 죽어도 자동화·관제는 NAS에서 계속 돈다.**

### 왜 이 구조가 목적 5(끊김 없음)를 유일하게 만족하나
- Claude Code 토큰 소진 → n8n은 LiteLLM(API 종량제)로 계속 동작. 영향 0.
- 스튜디오 다운 → LiteLLM 폴백체인이 저가 API로 승급. 자동화 무중단.
- 미니 다운 → 음성만 중단, 백본 무사. Uptime Kuma가 즉시 Slack 알림.
- NAS 다운 → 유일한 치명점. restic + docker-compose가 git에 있어 재구축 절차화(월 리허설).

## 3. 목적 → 구성요소 매핑

| 목적 | 실행 위치 | LLM 라우팅 | 비고 |
|---|---|---|---|
| 주식 자동매매 | **전용 코드 서비스(별도 컨테이너)@NAS — n8n 밖** | 리서치·스크리닝만 LLM, **주문 로직은 결정론적 코드+테스트** | 실돈 주문을 GUI 워크플로에 두지 않는다(v2.1 정정). n8n은 리포트 읽기만. 페이퍼 2주 → 소액 → 킬스위치·일손실한도 필수 |
| 수집→퀴즈·아카이빙 | n8n@NAS (RSS/크롤) → Postgres 아카이브 | 요약·퀴즈 = 저가 API 또는 local-large | 퀴즈는 Slack/보이스로 배달 |
| 링크드인 포스팅 | n8n@NAS + LinkedIn API | **대외 문안 = 프런티어 모델 고정** (T1 실측: 로컬모델 가격 10× 오류·사실 날조) | 발행 전 Slack 승인버튼 |
| 업무·구글 일정 | n8n 캘린더 노드 + 기존 google MCP | 저가 티어 | 기존 자산 재사용 |
| 리드 발굴→블로그→견적서 | n8n@NAS (스크래핑·보강) | 발굴·분류 = 저가/로컬, **블로그·견적서 = 프런티어** | 견적서는 승인 후 발송 |
| 멀티엔진 선택 | **LiteLLM 설정 파일이 곧 정책** | 작업클래스별 별칭: `external-doc`→claude, `bulk-summarize`→kimi/local, … | routing-policy.md를 코드화 |
| 영상 파이프라인 | n8n@NAS 오케스트레이션 | 키워드=트렌드API, 스크립트=프런티어, 이미지=스튜디오 로컬 or API, 렌더=Kling/Veo API, 더빙=TTS API | 렌더만 클라우드(기존 결론 유지) |
| 프로젝트 관리·관제 | Uptime Kuma(관제) + 기존 대시보드 확장 or Plane 셀프호스트@NAS | — | 2주 내엔 커스텀 제작 금지, 기존 대시보드 이관·확장 우선 |
| 어학 보이스 | voicebridge@미니 확장 | 대화=프런티어/로컬, 표현기록→Postgres@NAS | 기록→간격반복 퀴즈로 목적 2와 합류 |

## 4. 관제 설계 (끊김 없는 관제)

- **닻 = NAS Uptime Kuma**: 미니·스튜디오·각 컨테이너·핵심 플로우(heartbeat URL push)를 감시, 실패 시 Slack #agent-log. 맥이 다 죽어도 관제는 산다.
- 기존 agent-config heartbeat(맥 자체 점검·격리우회 감시)는 보완재로 유지.
- Slack 2채널 체계 유지(#agent 명령/승인 · #agent-log 관제). PM 도구는 기존 대시보드 확장(부족 판명 시에만 Plane 도입).

## 7. 적대적 자기검증 (v2.1 — 2026-07-22, v2 초안에서 발견된 결함과 수정)

v2 초안을 스스로 공격해 발견한 결함. 아래 수정이 본문에 반영됨:

1. **[정정] 주식 매매를 n8n에 뒀던 것** — 실돈 주문을 GUI 워크플로에 두는 건 부적절(디버깅·멱등성·재시도 반이중화 위험). → **전용 코드 서비스**(테스트·킬스위치·일손실한도 내장)로 분리, n8n은 읽기 전용.
2. **[사실확인] 링크드인 개인 프로필 자동 발행은 즉시 불가** — `w_member_social` 스코프는 **LinkedIn 앱 심사 필요(통상 2~4주, 최대 6~8주)**, 토큰 60일 만료, ~100콜/일 제한. → **D1에 API 신청**을 걸고, 승인 전까지는 "초안 자동생성 → Slack 승인 → 반자동 게시"로 운영. 계정 제재 위험이 있는 비공식 자동화(스크래핑 게시)는 금지.
3. **[정정] 2주 계획 과적** — 우선순위 명시: **P0** 백본+관제(D1~4) / **P1** 수집·퀴즈·일정·리드·견적(D5~9) / **P2** 매매 페이퍼모드·영상 MVP·어학 확장(D10~14, **밀리면 3~4주차로 이월 허용**). P0·P1 완성이 2주의 성공 기준.
4. **[자백] Claude 편향 잔존과 그 통제** — Claude Code(오케스트레이션 보조)·프런티어 기본값에 Claude를 남김. 통제장치: **백본(n8n+LiteLLM)은 Claude 의존 0** — Claude Code가 사라져도 자동화 무영향, 프런티어 슬롯은 설정 한 줄로 GPT/Gemini 교체 가능. 단 voicebridge는 claude_agent_sdk 의존(대화형이라 허용, 장기적으로 LiteLLM 경유 검토).
5. **[리스크] n8n 스프롤 = celery 재림 가능성** — 가드레일: 플로우 수 상한(12개), 핵심 로직은 git 관리 코드로, n8n 워크플로 JSON 주간 git 백업. 코드 우선 성향이 강해지면 Windmill(코드·git 네이티브)로 교체 검토.
6. **[리스크] UGOS Pro 미성숙** — 탈출로 확보: docker-compose를 호스트 불가지론적으로 작성. UGOS Docker가 불안정하면 **동일 compose를 미니의 기존 Docker로 이동**(미니에 docker 설치 확인됨) — 설계 무변경 이사.
7. **[단순화 옵션] LiteLLM 자가호스팅 부담** — 외부 엔진 묶음(OpenAI·Kimi 등)을 OpenRouter 한 키로 단순화 가능(소액 수수료). 로컬 ollama 티어 때문에 LiteLLM은 유지하되 업스트림을 OpenRouter 하나로 줄이는 구성 허용.
8. **[보강] 감시자의 감시자** — Uptime Kuma 자체가 죽으면? 미니/스튜디오의 기존 heartbeat가 Kuma URL을 역감시(상호 감시 쌍). 이미 있는 자산이라 비용 0.
