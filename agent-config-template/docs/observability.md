# 관제 설계 — 대시보드 다이어트와 에이전트 인벤토리

> 원칙: **대시보드를 늘려서 관제를 잘하는 게 아니라, 봐야 할 것을 4개로 줄여서 잘한다.**
> 현대 관제(SRE)의 핵심인 "골든 시그널"을 1인 운영 에이전트에 맞게 번역한 것이 아래 4개다.

## 1. 에이전트 골든 시그널 — 이 4개만 본다

| 시그널 | 질문 | 어디서 보나 | 자동화 |
|---|---|---|---|
| **생존** (uptime) | 데몬이 살아 있나? 크래시-재시작 루프는 아닌가? | #agent-log heartbeat + `[hermes] (재)시작됨` 알림 | heartbeat.sh (매일 8시) + hermes.plist 시작 래퍼 |
| **실패율** (errors) | 이번 주 :x: 알림 몇 건? 반복되나? | #agent-log 주간 리뷰 | guard.sh (즉시 알림, 실패 로그 보존) |
| **처리 흐름** (traffic) | 들어온 요청이 처리됐나? **유실은 없나?** | #agent 요청의 접수 ACK + #agent-log 작업 보고 | 에이전트 ACK 규칙(아래) + 주간 "ACK 없는 요청 스캔"(runbook 2번) |
| **변경** (change) | 시스템이 뭐로 바뀌었나? **무커밋 변조는 없나?** | git log + git-sync 알림 + heartbeat의 skills/ 드리프트 검사 | git-sync.sh + heartbeat.sh |

**ACK 규칙**: 에이전트는 #agent 채널의 요청을 받으면 즉시 접수 이모지(:eyes:) 또는 한 줄 응답을 남긴다.
"요청은 있는데 ACK가 없다" = 유실 의심 — 이것이 조용한 요청 증발을 잡는 유일한 그물이다.

이 4개 밖의 지표(토큰 사용량 그래프, 응답 지연 히스토그램 등)는 **문제가 생겼을 때만** 열어보는 참고 자료다. 상시 관제 대상이 아니다.

## 2. Mini 대시보드 다이어트 (UX 단순화)

증상: 헤르메스 대시보드에 메뉴가 너무 많아 활용이 벅참.
진단: 대시보드가 벅찬 것은 정상이다 — **pull형 관제를 전제로 만든 화면**이기 때문. 우리는 push형(알림이 옴)으로 전환했으므로 대시보드의 역할이 바뀐다.

### 처방: 3단계 다이어트

1. **역할 재정의**: 대시보드는 "매일 보는 화면"이 아니라 **"알림 받고 조사하러 들어가는 화면"**이다. 즐겨찾기/시작화면에서 빼도 된다. (조사용 원본 로그는 `~/agent-config/logs/`와 `~/Library/Logs/agent/`에도 남는다 — guard가 실패 전체 출력을 보존한다)
2. **메뉴 분류** — 대시보드의 각 메뉴를 아래 셋 중 하나로 표시한다:
   - **KEEP (3~5개)**: 로그 뷰어, 프로세스/연결 상태, 스킬 목록 — 조사에 실제로 쓰는 것
   - **OFF**: 끌 수 있는 기능은 끈다 (안 쓰는 채널 연동, 실험 기능, 통계 위젯)
   - **IGNORE**: 끌 수 없으면 안 본다고 결정한다 (결정도 관제 설계다)
3. **재점검은 월간 루틴에서**: runbook "매월 2번" 항목 (KEEP이 3~5개를 넘었으면 줄인다). 월간 작업 목록의 원본은 runbook이다 — 여기에 별도 주기를 만들지 않는다.

### 기록 (여기를 채워 커밋한다 — 이 표가 곧 대시보드 설정의 진실원)

| 메뉴/기능 | 분류 | 사유 |
|---|---|---|
| (예: 로그 뷰어) | KEEP | 실패 조사용 |
| (예: 텔레그램 연동) | OFF | Slack만 사용 |
| … | | |

## 3. 에이전트/채널 인벤토리 — 숨기기·끄기 기준

MVL 원칙: **기본값은 OFF.** 살아남으려면 아래 두 질문을 통과해야 한다.

1. 이게 고장나면 5분 안에 알 수 있는가?
2. 이것만 따로 끌 수 있는가?

### 인벤토리 (현행 워크스페이스 기준 — 상태를 바꾸면 이 표를 커밋한다)

| 에이전트/연동/채널 | 상태 | 사유 |
|---|---|---|
| Slack #agent (명령) | ON (신설 필요) | 유일한 명령 창구 |
| Slack #agent-log (관제) | ON (신설 필요) | 유일한 관제 창구 |
| #sns-biz (SNS 카피 검토 에이전트) | **OFF (안정화 기간)** | MVL 축소 대상. 2주 무사고 후 두 질문 통과 시 복원 |
| #biz-페이퍼크래프트사업 | 수동 전용 (PHASE 3에서 biz 전담 서브에이전트 채널로 전환 예정) | 전환 전 에이전트 명령은 #agent로 일원화 |
| #idea-사업타당성검토 | 수동 전용 | 에이전트 연동 대상 아님 |
| (예: 텔레그램 봇) | OFF | 채널 이원화 = 관제 구멍 |
| Mac Studio 모델 서버 | OFF (PHASE 2에서 health-check 훅과 함께 편입) | routing-policy 선행 작업 참조 |

- OFF로 바꾼 것은 삭제가 아니라 **비활성화**다. 필요해지면 두 질문을 통과시킨 뒤 되살린다.
- 이 표를 바꾸는 것도 커밋이다 — 무엇이 왜 꺼졌는지가 이력으로 남는다.

## 4. 개선 루프 (더 나은 결과를 계속 뽑는 방법)

주간 리뷰의 **개선 슬롯(runbook 3번, 15분)**에서 다음 셋 중 **하나만** 고른다 (전부 하려다 아무것도 안 하는 것 방지):

- 반복 실패 1개의 원인 제거, 또는
- 스킬 1개 개선 (자주 쓰는 것부터), 또는
- 수동으로 2회 이상 한 작업 1개를 스킬/훅으로 자동화

15분을 넘길 규모면 다음 주 슬롯의 계획으로 메모만 남긴다 (runbook과 동일 규칙).
이 "주 1개 개선"이 복리로 쌓이는 것이 고도화의 실체다. 한 번의 대공사보다 빠르다.
