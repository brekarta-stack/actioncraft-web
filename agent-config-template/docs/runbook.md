# 운영 런북 — 이 문서 밖의 것은 관리하지 않는다

> 총 부담: **매일 10초 + 주 30분 + 월 1회 30분.**
> 이 범위를 넘는 관리가 필요해지면, 그것은 운영자가 부지런해질 일이 아니라
> 시스템에서 기능을 빼야 한다는 신호다.

## 매일 (10초) — 아침 커피와 함께

1. Slack `#agent-log` 열기.
2. 오늘 아침 8시 `[heartbeat]` 메시지 확인:
   - :green_heart: → 끝. 아무것도 하지 않는다.
   - :rotating_light: (hermes=DOWN) → Mini에서 `launchctl list | grep com.agent` 확인, 메시지 안 안내대로 조치.
   - **heartbeat 자체가 안 옴** → Mini가 꺼졌거나 네트워크 단절. Mini 전원·네트워크 확인.

## 매주 (30분) — 요일을 정해 고정 (권장: 금요일 오후)

1. **스킬 심사 (15분)**: `skills/AUDIT.md` 체크리스트로 `_quarantine/` 처리.
2. **로그 리뷰 (10분)**: 지난 주 `#agent-log`의 `:x:` 알림을 훑고,
   - 같은 실패가 2회 이상 반복 → 원인 메모를 남기고 수정 계획 (반복 실패는 방치 금지)
   - 실패 0회면 그냥 끝.
3. **커밋 확인 (5분)**: `git log --oneline -10` 으로 이번 주 변경 이력 확인, 미커밋 변경이 있으면 커밋·푸시.

## 매월 (30분) — 복구 리허설

- `docs/recovery.md` 절차를 그대로 수행한다 (별도 폴더에 clone → 복구되는지 확인).
- 통과하면 `#agent-log`에 "recovery drill OK (날짜)" 한 줄 남긴다.
- 실패하면 그것이 이번 달 최우선 수리 항목이다.

## 확장을 결정할 때 (수시)

새 기능·채널·모델을 추가하기 전에 두 질문:

1. 이게 고장나면 **5분 안에 알 수 있는가?** (없으면 guard.sh/알림부터)
2. 이것만 **따로 끌 수 있는가?** (없으면 설계를 바꾼다)

두 질문을 통과 못 하면 추가하지 않는다.
